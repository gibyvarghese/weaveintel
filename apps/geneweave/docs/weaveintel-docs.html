Building applications with weaveIntel

weaveIntel is a TypeScript framework for production-grade AI applications. It gives you composable, vendor-neutral building blocks — from a single chat completion to a fleet of long-running, self-coordinating agents — with tool calling, governance, observability and resilience woven into the fabric rather than bolted on afterward.

This guide is written for adopters: engineers who want to assemble their own product on top of weaveIntel rather than read API reference in isolation. For every package you'll find the same shape of answer — what it does, why and when you'd reach for it, how to wire it up, and what each option means — followed by a copy-ready snippet. The packages are grouped into parent structures that mirror how a real application is layered, so you can read top-to-bottom or jump straight to the layer you're working on.

📌
A note on accuracy

Every signature and option in this guide is taken directly from the package source and the runnable examples in the repository, and the snippets have been executed against the library to confirm they run. Where a call takes an execution context, it is the first argument throughout.

What you can build

The framework is deliberately wide because real AI products need more than a model call. A typical weaveIntel application stitches together a handful of layers:

  • A model layer that is provider-neutral — you write against one Model interface and swap OpenAI, Anthropic, Google, or a local model behind it, optionally letting a router choose per request.
  • An agent layer with two complementary runtimes: weaveAgent for request/response ReAct loops, and weaveLiveAgent for agents that live for days, wake on a schedule, accumulate state, and coordinate in a mesh.
  • A tool layer where every tool is a governed asset — it carries a risk class, flows through a policy gate, an approval gate, a rate limiter and an audit trail, and shares one resilience pipeline.
  • A knowledge layer — chunking, embeddings, retrieval, three flavours of memory, a knowledge graph, and a semantic cache.
  • A safety & governance layer — guardrails, PII redaction, per-tenant encryption, identity and access control, budgets, compliance and sandboxing.
  • An observability layer — every model call, tool call and agent step emits a span with token usage and cost; runs can be replayed from their trace.

You don't have to adopt all of it. The core is just contracts and types with zero vendor dependencies; everything else is an implementation you opt into. Start with a model and an agent, and add layers as the product grows.

How the framework is organised

There are roughly eighty packages, but they fall into a small number of families. The sidebar is grouped the same way. The single most important architectural rule to internalise is this: interfaces live in @weaveintel/core; implementations live in leaf packages. Core never imports from a provider or a runtime — the dependency arrow always points toward core. That is what makes the framework vendor-neutral and what lets you replace any layer without touching the others.

Getting started #

You need Node.js ≥ 20, npm ≥ 10, and at least one provider key if you want to talk to a real model. The repository is an npm-workspaces monorepo built with Turborepo.

bash
# clone, install, and build every package (dependency-aware)
git clone https://github.com/gibyvarghese/weaveintel.git
cd weaveintel
npm install
npm run build

# provide at least one provider key
export OPENAI_API_KEY=sk-...
export ANTHROPIC_API_KEY=sk-ant-...

# run any example directly with tsx — most need no API key
npx tsx examples/02-tool-calling-agent.ts   # ReAct agent, no key needed
npx tsx examples/01-simple-chat.ts          # needs OPENAI_API_KEY
npx tsx examples/52-live-agents-basic.ts    # a long-running live agent

The fastest way to see everything wired together is the reference application, geneWeave, which is built only from @weaveintel/* packages and run with ./scripts/start-geneweave.sh. But you do not need it to use the library — the snippets throughout this guide are self-contained.

Your first program: a tool-calling agent

This is the smallest interesting weaveIntel program. It creates an execution context, registers one tool, builds a ReAct agent over a model, and runs it. Two signatures to internalise from the start: the model factory takes the model id first (weaveOpenAIModel('gpt-4o-mini', { apiKey })), and agent.run takes the context first (agent.run(ctx, input)).

first-agent.ts
import { weaveContext, weaveToolRegistry, weaveTool } from '@weaveintel/core';
import { weaveAgent } from '@weaveintel/agents';
import { weaveOpenAIModel } from '@weaveintel/provider-openai';

// 1. A registry holds the tools the agent is allowed to call.
const tools = weaveToolRegistry();
tools.register(weaveTool({
  name: 'get_weather',
  description: 'Get the current weather for a city',
  parameters: { type: 'object', properties: { city: { type: 'string' } }, required: ['city'] },
  execute: async ({ city }) => `${city}: 22°C, sunny`,
}));

// 2. A model — provider-neutral behind the Model interface.
const model = weaveOpenAIModel('gpt-4o-mini', { apiKey: process.env.OPENAI_API_KEY });

// 3. The agent wires model + tools into a ReAct loop.
const agent = weaveAgent({ model, tools, systemPrompt: 'You are a weather assistant.', maxSteps: 5 });

// 4. Run it. Context first, then the input messages.
const ctx = weaveContext({ userId: 'demo' });
const result = await agent.run(ctx, { messages: [{ role: 'user', content: 'Weather in Paris?' }] });
console.log(result.output);   // → the model's final answer

That program already benefits from things you didn't write: if a runtime with guardrails is attached to the context, the tool call and the final answer are checked; every step can emit a trace span; and the model call goes through a shared resilience pipeline. Those are the framework's defaults, covered throughout this guide and summarised in Security & defaults.

The mental model #

Five concepts are enough to navigate the entire framework. Learn these and the rest is composition.

Model

An adapter to an LLM that implements one Model interface — generate(ctx, request) and optional stream, embeddings, vision, structured output. Swapping vendors means swapping an import.

Tool

A typed function an agent can call. It has a JSON-schema, a risk class, and passes through policy and resilience. Tools may come from local code, an MCP server, or another agent — all through the same contract.

Agent

An LLM-driven loop that picks tools to satisfy a goal. weaveAgent is one-shot; weaveLiveAgent is long-running. They share option names so knowing one teaches the other.

Mesh

A group of live agents sharing a state store, a heartbeat schedule, and a message bus. A mesh is a team; agents inside it exchange messages and file evidence as contracts.

Policy / Contract

Declarative rules for what tools may do, when a human must approve, and what evidence an agent must produce before a task counts as done.

Context

The ExecutionContext threads userId, tenantId, a trace id, a budget, a deadline, a cancellation signal, and the ambient runtime through every call. It is how multi-tenancy, observability and cancellation work without extra plumbing.

Everything in weaveIntel is shaped by one principle: cross-cutting concerns travel on the context, not in your call sites. You don't reach for process.env, you don't construct a tracer per call, and you don't sprinkle retry logic — those live on the runtime carried by the context, and the framework consults them for you.

Naming & conventions

The codebase follows two prefixes consistently, and learning them tells you what a symbol is before you read its docs:

  • weaveX(...) — a user-facing constructor that returns a runnable thing you instantiate: an agent, a model, a store, a tool, a tracer. If you're building an app, these are your entry points.
  • createX(...) — an internal factory for plumbing: a registry, a dispatcher, a scheduler, a policy-enforced wrapper. You use these when assembling infrastructure rather than calling models.

A few more conventions worth knowing up front. IDs that may be persisted are minted with time-ordered UUIDv7 (newUUIDv7()) rather than random UUIDs. Outbound HTTP goes through a hardened, SSRF-checked fetch rather than raw fetch. And the framework is "DB-driven, not code-driven" wherever it can be: tool catalogs, policies, prompts, skills, models and routing are designed to live in database rows that admin UIs and runtime resolvers read, so product behaviour can change without a code deploy.

The curated entry point

Most adopters need roughly eight packages out of the ~78 the monorepo ships, and importing them one by one is an onboarding cliff. @weaveintel/weaveintel is a meta-package that re-exports the golden-path surface — the runtime, execution context, hardened egress, secret resolution, audit, the agent and tool constructors — behind one import. There is no hidden behaviour in it, only re-exports; anything beyond the curated set still lives in its specific package.

// One import for the common case, instead of reaching into many packages.
import { weaveRuntime, weaveContext, weaveAgent, defineTool } from '@weaveintel/weaveintel';

@weaveintel/core — contracts & the runtime #

@weaveintel/corezero vendor deps

What it does. Core is the spine of the framework: it defines every interface, type and shared runtime utility, and it depends on nothing. Providers, tools, agents and the rest all import from core; core imports from none of them. When you read core's index.ts you are reading the whole contract surface of weaveIntel in one place — model contracts, tool contracts, agent contracts, memory, retrieval, guardrails, human-tasks, workflows, routing, tenancy, encryption, observability and more, each as pure types plus a small number of constructors.

Why it matters to you. Because the contracts live here and only here, you can build against weaveIntel without committing to any vendor or implementation. If you write a function that accepts a Model, it works with OpenAI, Anthropic, a local Ollama model, or a fake you wrote for tests — they all implement the same interface from core. The same is true of Tool, MemoryStore, VectorStore, Tracer and every other capability. This is the practical payoff of "interfaces in core, implementations in leaves": your application code is portable across the entire ecosystem.

Core also ships a handful of runtime utilities that are not just types, because some cross-cutting behaviour must be shared rather than re-implemented. The most important are the execution context, the ambient runtime, the tool factory, the hardened egress client, the SSRF guard, the UUIDv7 minter, the error classifier and the event bus. The rest of this section covers the three you touch most.

ExecutionContext — the thread through everything

Every operation in weaveIntel takes an ExecutionContext as its first argument. Rather than passing ten parameters down a call stack, the context carries identity, cancellation, deadlines, trace continuity, request-scoped metadata, a budget, and the ambient runtime. You create one with weaveContext(...) and derive scoped children with weaveChildContext(...).

FieldMeaning
executionIdUnique id for this execution; the trace root. Auto-generated if you don't supply one.
tenantIdTenant isolation key. Flows into config resolution, budgets, encryption and audit.
userIdUser identity for access control and per-user audit records.
signalAn AbortSignal for cooperative cancellation/timeout.
deadlineEpoch-ms deadline. isExpired(ctx) and deadlineSignal(ctx) turn it into enforcement.
parentSpanIdLinks spans together for trace continuity across nested calls.
tracerOptional tracer for automatic span emission. Inherited from the runtime if not set explicitly.
runtimeThe ambient WeaveRuntime — egress, tracer, secrets, audit, persistence, resilience, guardrails.
metadataArbitrary request-scoped key/values, merged (not replaced) when you derive a child.
budgetAn ExecutionBudget: maxTokens, maxCostUsd, maxSteps, maxDurationMs, maxRetries.
context.ts
import { weaveContext, weaveChildContext, isExpired, deadlineSignal } from '@weaveintel/core';

const ctx = weaveContext({
  userId: 'u_123',
  tenantId: 'acme-corp',
  deadline: Date.now() + 30_000,                 // 30s hard ceiling
  budget: { maxSteps: 8, maxCostUsd: 0.25 },     // per-execution budget
  metadata: { feature: 'support-bot' },
});

// Derive a scoped child for a sub-operation; metadata merges, signal is inherited.
const child = weaveChildContext(ctx, { metadata: { step: 'retrieval' } });

if (isExpired(child)) throw new Error('deadline exceeded');
const signal = deadlineSignal(child);            // AbortSignal that fires at the deadline

Default outcome — observability rides the context

If a tracer is present on the runtime, weaveContext picks it up automatically. Downstream agents and tools call weaveResolveTracer(ctx) and emit spans without any extra wiring from you — token usage, duration and cost end up on the trace for free.

The ambient runtime — egress, secrets, audit, guardrails

What it is. The WeaveRuntime is a single object bundling the cross-cutting concerns no call site should construct on its own: a hardened egress client, a tracer, a secret resolver, an audit logger, a persistence slot, a resilience slot, and a guardrails slot. You build one with weaveRuntime(...) and attach it to your context. From then on, features that declare they require a capability are checked against it.

Why use it. Three reasons. First, secrets: packages resolve credentials through runtime.secrets rather than reading process.env directly, so a vault or per-tenant override can be plugged in without touching any call site. Second, egress safety: all outbound HTTP flows through one SSRF-checked, timeout-bounded, size-capped client instead of thirteen hand-rolled fetch wrappers. Third, uniform policy: when guardrails live on the runtime, every agent in the process enforces the same input/output and tool-call checks — you cannot accidentally ship one agent that bypasses them.

SlotPurpose
egressHardened fetch + streaming with SSRF validation, redirect re-checking, timeouts and size caps.
tracerSpan emitter used across runtimes; auto-propagates onto every context derived from this runtime.
secretsSecretResolver — env, in-memory, or a chain. Use requireSecret to fail fast on a missing key.
auditDurable, optionally-redacting audit logger. Agents and tools emit weaveAudit(ctx, …) records automatically.
persistenceA KV/persistence slot (in-memory by default) that durable subsystems write to.
resilienceThe shared rate-limit + circuit-breaker + retry pipeline (see Resilience).
guardrailsOptional checkToolCall / checkOutput hooks; consulted by agents fail-closed (see Guardrails).
runtime.ts
import {
  weaveRuntime, weaveContext, envSecretResolver,
  weaveInMemoryPersistence, RuntimeCapabilities,
} from '@weaveintel/core';
import { weaveInMemoryTracer } from '@weaveintel/observability';

const runtime = weaveRuntime({
  tracer: weaveInMemoryTracer(),
  secrets: envSecretResolver(),               // resolve via runtime.secrets, not process.env
  persistence: weaveInMemoryPersistence(),
  // guardrails: yourGuardrailsSlot,  // optional — see Guardrails section
});

// Attach to every context; the tracer propagates automatically.
const ctx = weaveContext({ userId: 'u_1', runtime });

// Tools/features can declare they need a capability; the runtime asserts it.
runtime.require(RuntimeCapabilities.NetEgress);   // throws early if egress is absent
🔐
Default outcome — safe egress & secret hygiene

The hardened egress client blocks requests to cloud metadata endpoints, RFC-1918 private ranges and DNS-rebinding bypasses, enforces an HTTPS/TLS floor (plain http:// only for loopback when explicitly allowed), follows redirects manually so each hop is re-validated, and caps response size and wall-clock time. Resolving secrets through the runtime means a credential never has to be read from the environment inside a leaf package — a prerequisite for per-tenant vaults.

Defining tools with weaveTool

A tool is the primary way you extend an agent. weaveTool(...) turns a plain async function into a Tool that conforms to the framework's contract: a name and description the model sees, a JSON-schema for the arguments, an execute function, and metadata that governs how the tool is allowed to run. weaveToolRegistry() holds a set of tools and is what you hand to an agent.

OptionMeaning
nameStable identifier the model emits to call the tool. Also the audit / policy key.
descriptionNatural-language summary the model uses to decide when to call it. Write it for the model, not for humans.
parametersA JSON-schema object describing the arguments. The model is constrained to produce arguments matching this shape.
executeasync (args, ctx) => string | ToolOutput. Return a string for a simple result, or a ToolOutput with isError and metadata for richer cases.
riskLevelOne of read-only · write · destructive · privileged · financial · external-side-effect. Drives the policy risk gate. Defaults to read-only.
requiresApprovalWhen true, the governed registry creates an approval request and refuses to run until a human resolves it.
tagsFree-form labels; registry.listByTag(tag) filters on them.
requiresRuntime capabilities the tool needs (e.g. runtime.net.egress). Asserted before invocation — there is no silent bypass.
define-tool.ts
import { weaveTool, weaveToolRegistry, RuntimeCapabilities } from '@weaveintel/core';

const registry = weaveToolRegistry();

registry.register(weaveTool({
  name: 'refund_order',
  description: 'Issue a refund for an order by id',
  parameters: {
    type: 'object',
    properties: {
      orderId: { type: 'string' },
      amountUsd: { type: 'number', description: 'Amount to refund' },
    },
    required: ['orderId', 'amountUsd'],
  },
  riskLevel: 'financial',          // moves it behind the risk + approval gates
  requiresApproval: true,           // a human must approve each call
  requires: [RuntimeCapabilities.NetEgress],
  execute: async ({ orderId, amountUsd }, ctx) => {
    // ctx carries tenantId/userId/budget/deadline for you to honour
    return { content: `Refunded $${amountUsd} on ${orderId}` };
  },
}));

A tool created this way is already governable: the moment you place it in a policy-enforced registry, its riskLevel is checked against the tenant's allowed levels, requiresApproval routes it to an approval queue, and every invocation is audited. You write the function once; the governance is applied by the registry, not by you.

Models & providers #

The model layer is where weaveIntel's vendor-neutrality is most tangible. There is exactly one Model interface, and every provider implements it. Your application talks to model.generate(ctx, request) (and optionally model.stream(ctx, request)) and never knows which vendor is behind it.

@weaveintel/models — the unified router

@weaveintel/models

What it does. This package provides a provider registry and a unified constructor, weaveModel(...), that builds a model from a provider name plus a model id, with optional automatic fallback, middleware and an event bus. Provider packages register themselves at import time, so importing @weaveintel/provider-openai makes 'openai' available to the router.

Why use it instead of a provider factory directly. Two reasons. You get fallback — if the primary endpoint is unhealthy or errors, the router transparently tries the next candidate. And you get capability selection — weaveSelectModel can pick a registered model that satisfies a capability (vision, structured output, embeddings) so you don't hardcode model ids in business logic.

Option (CreateModelOptions)Meaning
providerRegistered provider key, e.g. 'openai', 'anthropic', 'google', 'ollama'.
modelThe provider-specific model id, e.g. 'gpt-4o-mini' or 'claude-sonnet-4-20250514'.
optionsProvider options bag (apiKey, baseUrl, headers…). Passed straight to the provider factory.
fallbackAn array of further CreateModelOptions tried in order when the primary fails.
middlewareRequest/response middleware (e.g. observability) wrapped around every call.
eventBusEvent bus to publish model lifecycle events on.
router.ts
import { weaveModel } from '@weaveintel/models';
import '@weaveintel/provider-anthropic';   // registers 'anthropic'
import '@weaveintel/provider-openai';      // registers 'openai'

const model = weaveModel({
  provider: 'anthropic',
  model: 'claude-sonnet-4-20250514',
  options: { apiKey: process.env.ANTHROPIC_API_KEY },
  fallback: [{ provider: 'openai', model: 'gpt-4o-mini', options: { apiKey: process.env.OPENAI_API_KEY } }],
});

const res = await model.generate(ctx, { messages: [{ role: 'user', content: 'Capital of France?' }] });
console.log(res.content);   // "Paris" — from primary, or from the fallback if the primary failed

Provider adapters

Each provider package is a thin adapter implementing the core Model (and where relevant EmbeddingModel, ImageModel, AudioModel, ModerationModel, ResponseModel) interfaces. They all follow the same constructor shape: model id first, options second.

@weaveintel/provider-openai
Chat, embeddings, image generation, audio (TTS/STT), the Responses API agentic loop, managed vector stores, file storage, moderation and fine-tuning. The broadest adapter.
@weaveintel/provider-anthropic
Chat, streaming, tool use, vision, PDF input, extended thinking, citations, message batches, and computer use.
@weaveintel/provider-google
Google Gemini (1.5 / 2.5 / 3.x): chat, tool use, vision and streaming.
@weaveintel/provider-ollama
Local LLMs through Ollama — Llama, Mistral, Qwen, Phi, Gemma, DeepSeek and more — plus local embeddings.
@weaveintel/provider-llamacpp
Local GGUF models via a llama.cpp HTTP server through its OpenAI-compatible API.

How to use. Construct directly when you want a pinned model, or register them and use the router for fallback. The chat call shape — generate(ctx, request) with a messages array, optional tools, toolChoice, responseFormat, temperature, maxTokens, topP, stop — is identical across providers. Streaming yields StreamChunks of type text, tool_call and usage.

providers.ts
import { weaveContext } from '@weaveintel/core';
import { weaveOpenAIModel, weaveOpenAIEmbeddingModel } from '@weaveintel/provider-openai';
import { weaveAnthropicModel } from '@weaveintel/provider-anthropic';
import { weaveOllamaModel } from '@weaveintel/provider-ollama';

const ctx = weaveContext({ userId: 'u_1' });

// Same shape everywhere — model id first, options second.
const gpt    = weaveOpenAIModel('gpt-4o-mini', { apiKey: process.env.OPENAI_API_KEY });
const claude = weaveAnthropicModel('claude-sonnet-4-20250514', { apiKey: process.env.ANTHROPIC_API_KEY });
const local  = weaveOllamaModel('llama3.1', { baseUrl: 'http://localhost:11434' });

// Structured output via responseFormat: 'json_schema'
const structured = await gpt.generate(ctx, {
  messages: [{ role: 'user', content: 'Extract name and age: "Ada, 36"' }],
  responseFormat: { type: 'json_schema', name: 'person',
    schema: { type: 'object', properties: { name: { type: 'string' }, age: { type: 'number' } } } },
});

// Streaming — iterate StreamChunks (text / tool_call / usage)
for await (const chunk of claude.stream!(ctx, { messages: [{ role: 'user', content: 'Tell me a haiku' }] })) {
  if (chunk.type === 'text') process.stdout.write(chunk.text);
}

// Embeddings
const emb = weaveOpenAIEmbeddingModel('text-embedding-3-small', { apiKey: process.env.OPENAI_API_KEY });

Default outcome — every provider call is resilient

All three hosted providers wrap their HTTP calls in the shared resilience pipeline with PROVIDER_RESILIENCE_DEFAULTS: two auto-retries with 500 ms base / 30 s cap and jitter, and a circuit breaker that opens after eight consecutive failures with a 30 s cooldown. A 429 anywhere becomes a normalized signal the rest of your app can react to. Custom providers are expected to reuse the same constant.

@weaveintel/routing — smart, explainable routing

@weaveintel/routing

What it does. Where the models router gives you simple fallback, routing gives you a full decision engine. The SmartModelRouter takes a set of candidate models with cost, quality and capability metadata, tracks each endpoint's health, scores them with a weighted strategy, and returns a RoutingDecision that explains itself — you can see why a model was chosen. It also infers a task type from the request and can route per task (cheap model for classification, strong model for code).

When to use it. Reach for routing when model choice is a product concern rather than a constant: when you want to spend less on easy turns, when different tenants get different model tiers, or when you're running A/B experiments and want a feedback loop from outcomes back into the scorer. Because decisions are explainable and stored, you can audit and tune them.

Building blockRole
SmartModelRouterThe router. router.route(ctx, request, candidates) returns a scored, explained decision.
ModelHealthTrackerRecords success/latency/error per endpoint so unhealthy models are de-prioritised.
ModelScorerCombines cost, quality and capability fit into a weighted score.
inferTaskTypeDerives a task type (e.g. code, summarise, classify) from the request to drive task-aware routing.
InMemoryDecisionStorePersists decisions for audit, A/B analysis and the feedback loop.
smart-routing.ts
import { SmartModelRouter } from '@weaveintel/routing';

const router = new SmartModelRouter({ /* policies, scorer, health tracker */ });

const decision = await router.route(ctx, request, candidates);
console.log(decision.selectedModel, decision.reason);   // chosen model + human-readable rationale

Routing is the engine behind geneWeave's cost-by-task dashboard and routing experiments; the runnable examples 14-smart-routing.ts, 70-task-aware-routing.ts and 72–74 walk through health, task inference, the admin CRUD and the production feedback loop.

@weaveintel/testing — fakes for fast, key-free tests

@weaveintel/testing

What it does. Provides drop-in fakes that implement the core interfaces: a fake model, a fake embedding model, a fake vector store, and fake MCP transports. Because they satisfy the same contracts as real implementations, you can build and test an entire agent, RAG pipeline or mesh with no network and no API key.

Why it matters. Most of the ~150 examples in the repo run with no key at all precisely because they use these fakes. In your own test suite, swap the real model for weaveFakeModel and assert on the agent's steps deterministically.

testing.ts
import { weaveFakeModel, weaveFakeEmbedding, weaveFakeVectorStore } from '@weaveintel/testing';

const model = weaveFakeModel({ /* scripted responses for deterministic tests */ });
const embed = weaveFakeEmbedding();
const store = weaveFakeVectorStore();
// build your agent/RAG over these exactly as you would over real implementations

Agents & orchestration #

weaveIntel gives you two agent runtimes that share a vocabulary. weaveAgent is the one-shot ReAct agent you reach for first; weaveLiveAgent is its long-running twin for agents that live across time and coordinate in a mesh. Around them sit workflows for deterministic multi-step processes, human-in-the-loop tasks for approvals, completion contracts for evidence, and recipes that pre-assemble common patterns.

@weaveintel/agents — weaveAgent, the ReAct loop

@weaveintel/agents

What it does. weaveAgent implements the classic ReAct loop: it sends the conversation to the model; if the model returns tool calls it executes each tool, appends the results, and loops; when the model returns plain text, that text is the answer. It tracks steps and token usage, enforces budgets and deadlines, supports streaming, and consults guardrails on both tool calls and the final output.

When to use it. Any request/response interaction: a chat turn, a "do this task and tell me the result" call, a tool-using assistant. The user is waiting for an answer and the agent runs in-process. If the agent instead needs to run for hours, wake on a schedule, or remember across sessions, that is weaveLiveAgent's job.

OptionMeaning
modelThe Model the agent reasons with. Required.
toolsA ToolRegistry the agent may call. Its definitions are sent to the model so it knows what's available.
systemPromptInstructions injected as the system message at the top of the conversation.
maxStepsMaximum think→act loops before stopping. Defaults to 20 (30 in supervisor mode). A safety valve against runaway loops.
busAn EventBus that receives run/step/tool lifecycle events for observability.
memoryAn AgentMemory whose history is loaded before the run and appended to afterward.
policyAn AgentPolicy with shouldContinue and approveToolCall hooks — a per-agent governance escape hatch.
workersWorker definitions. Supplying these flips the agent into supervisor mode (see below).
nameAgent name used in events and audit records.

The run methods are agent.run(ctx, input) for a complete result and agent.runStream(ctx, input) for an async iterable of step events (text_chunk, tool_start, tool_end, step_end, done). The input is { messages } (and optionally a goal). The result carries output (final text), steps (each think/act event), usage (tokens, steps, tool calls, duration) and status (completed · failed · cancelled · budget_exceeded).

weave-agent.ts
import { weaveContext, weaveToolRegistry, weaveTool, weaveEventBus } from '@weaveintel/core';
import { weaveAgent } from '@weaveintel/agents';
import { weaveOpenAIModel } from '@weaveintel/provider-openai';

const tools = weaveToolRegistry();
tools.register(weaveTool({

@weaveintel/live-agents — long-running agents & meshes #

@weaveintel/live-agents12k+ LOC6 state backends

What it does. Live agents are the temporal extension of weaveAgent. Where a one-shot agent answers and exits, a live agent persists: it lives as a row in a state store, wakes on a heartbeat tick, decides what to pay attention to, takes one action, files evidence, and goes back to sleep. Groups of live agents form a mesh — a team that shares a state store, exchanges messages, promotes work through stages, and can bridge to other meshes under mutual approval.

When to use it. When the agent's lifetime is longer than a request. A market monitor that watches prices for weeks. A research assistant that accumulates findings over days. A Kaggle competitor that iterates on submissions. A support coordinator that processes an inbound queue on a schedule. Anything that must wake on a clock or an external event, carry state forward, and coordinate with peers belongs here.

The pieces and how they fit

Five primitives make up the runtime. Understanding their relationship is the key to the package:

  • Mesh — the team. It has an id, a tenant, a charter, and a list of actions that require dual control. Agents belong to a mesh.
  • LiveAgent — a persistent agent row: id, mesh, name, role, the id of its current contract, and status.
  • AgentContract — the agent's charter as data: its persona, objectives, success indicators, budget caps, working-hours schedule, attention-policy reference, review cadence and context policy. Contracts are versioned and immutable.
  • StateStore — the durable substrate. Everything (meshes, agents, contracts, messages, ticks, promotions, grants, accounts, bridges) is saved and loaded through it. Six backends ship.
  • Heartbeat — the scheduler. On each tick(ctx) it claims due ticks, asks the agent's AttentionPolicy what to do, and runs that action through an ActionExecutor.

The flow on every tick is: the heartbeat claims a scheduled tick → loads the agent and its contract → the attention policy returns an AttentionAction (e.g. "process message X", "start a task", "do nothing") → the action executor carries it out → the outcome is written back onto the tick. That is the whole loop. Everything else — contracts, promotions, grants, bridges — is structure layered on top of it.

weaveLiveAgent — the canonical constructor

Just as weaveAgent is the friendly front door to the ReAct loop, weaveLiveAgent is the friendly front door to a live agent's per-tick LLM behaviour. It deliberately mirrors weaveAgent's option names — name, model, tools, systemPrompt, maxSteps, policy — so knowing one teaches the other. It returns { handler, definition }: a TaskHandler you register against the runtime, and a declarative description of which capability slots you supplied.

Contracts, promotions, grants & bridges — the governance fabric

Live agents earn their "autonomous but accountable" character from four data structures, all persisted in the state store:

  • Contracts are the agent's immutable charter and its evidence ledger. An agent files findings as new contract versions; nothing is mutated in place, so you always have an audit trail of what the agent believed and when.
  • Promotions move work through stages with explicit requests and approvals — the framework refuses self-promotion (SelfPromotionForbiddenError).
  • Capability grants let one party hand a capability to another, but an agent can never self-grant (SelfGrantForbiddenError); only humans bind external accounts (OnlyHumansMayBindAccountsError). Agents inherit capabilities, they never mint them.
  • Cross-mesh bridges route contracts between teams, and any action listed in a mesh's dualControlRequiredFor needs two-party approval before it proceeds.
🔐
Default outcome — humans hold the keys

The error types are not decoration; they are enforced invariants. An agent cannot promote its own work, cannot grant itself a capability, and cannot bind an external account (Gmail, Slack, a broker). These are the structural guarantees that make a long-running autonomous mesh safe to run unattended.

@weaveintel/workflows — deterministic multi-step processes

@weaveintel/workflows12k+ LOC

What it does. Agents are good at open-ended reasoning; workflows are good at processes you want to be deterministic, durable and replayable. The workflow engine runs a graph of typed steps with branching, loops, parallelism, sub-workflows, human-task gates, dynamic expansion, checkpointing and compensation. You author the graph with a fluent builder and execute it with an engine that persists checkpoints so a crashed run resumes where it left off.

When to use it over an agent. When the sequence is known and you want guarantees: an onboarding pipeline, a content publication flow with a quality gate, an order-fulfilment saga with compensation on failure, a nightly data process. Steps can themselves be agentic, so a workflow can call an agent at one node and run plain code at the next — you get determinism for the spine and intelligence where you need it.

The builder (defineWorkflow(name)) exposes one fluent method per step type, each chaining and returning the builder; .build() freezes it into an immutable definition.

.agentic(id, name, …)Runs an agent at this node — intelligence inside a deterministic spine.
.condition(id, name, { handler, trueBranch, falseBranch })Boolean branch on the handler's return.
.branch / .switchMulti-way branching to one of several named steps.
.forEach / .fork / .joinIterate a collection; fan out parallel branches and re-join them.

.wait(id, name, { wakeAfterMs })Durable sleep — the run is checkpointed and resumes after the delay.

  .condition('gate', 'Quality Gate', { handler: 'qualityGate', trueBranch: 'publish', falseBranch: 'revise' })
  .deterministic('revise',  'Revise',  { handler: 'reviseContent', next: 'review' })
  .deterministic('publish', 'Publish', { handler: 'publishContent', next: 'notify' })
  .deterministic('notify',  'Notify',  { handler: 'sendNotifications' })

engine.registerHandler('editorialReview', async (vars) => ({ reviewed: true }));

engine.registerHandler('reviseContent',   async (vars) => ({ draft: '…' }));
engine.registerHandler('publishContent',  async (vars) => ({ url: '…' }));
engine.registerHandler('sendNotifications', async (vars) => ({ sent: true }));

@weaveintel/tools — the governed tool registry

@weaveintel/tools

What it does. Wraps an ordinary ToolRegistry so that every invocation passes through a six-stage pipeline before the tool's code ever runs. You build it with createPolicyEnforcedRegistry(...) (or wrap a single tool with createPolicyEnforcedTool), supplying a policy resolver, an approval gate, a rate limiter and an audit emitter. The resolver can be in-memory for tests or DB-backed in production, so policy lives in data, not code.

Why this is the important part. It is where weaveIntel's "governance is woven in" claim becomes concrete. Your tool author writes execute; the registry decides whether this caller, in this tenant, with this risk level, under this rate limit, with this approval state, is allowed to run it — and records the outcome either way. The pipeline, in order:

  1. Enabled check — a disabled tool is refused outright.
  2. Circuit breaker — if the tool's circuit is open from prior failures, it is short-circuited.
  3. Risk-level gate — the tool's declared riskLevel must be within the tenant/policy's allowedRiskLevels.
  4. Approval gate — if the policy marks it approval-required, an approval request is created and the call blocks until resolved.
  5. Rate limit — per-tool, per-scope rate limiting (e.g. per chat).
  6. Execute + audit — runs through the shared resilient callable with an optional timeout, then emits an audit event with the outcome (and input/output if the policy permits logging them).

The default policy, DEFAULT_TOOL_POLICY, is permissive (risk read-only, all six risk levels allowed) so nothing breaks out of the box; you tighten it per tenant or per tool by supplying a resolver.

governed-tools.ts
import {
  createPolicyEnforcedRegistry, InMemoryToolPolicyResolver,
  DEFAULT_TOOL_POLICY, noopAuditEmitter,
} from '@weaveintel/tools';

// The resolver takes a Map of toolName → partial policy (not a plain object).
const resolver = new InMemoryToolPolicyResolver(new Map([
  ['refund_order', { ...DEFAULT_TOOL_POLICY, allowedRiskLevels: ['read-only', 'financial'],
                  requiresApproval: true, rateLimitPerMinute: 5 }],
]));

const governed = createPolicyEnforcedRegistry(rawRegistry, {
  resolver,
  approvalGate,          // 4: blocks until a human approves
  rateLimiter,           // 5: per-tool/per-scope limiting
  auditEmitter: noopAuditEmitter,   // 6: a constant (replace with a durable emitter in prod)
  resolutionContext: { chatId: 'chat-1', userId: 'u_1', agentPersona: 'support' },
});

// Hand the GOVERNED registry to the agent — same interface, now gated end-to-end.
const agent = weaveAgent({ model, tools: governed });
🔐
Default outcome — every tool call is gated and audited

Wrap once, after all tools are registered, and every call is checked through the six stages and recorded — whether it succeeds, is denied, times out or trips the breaker. A ToolPolicyViolationError carries the exact reason (disabled, circuit_open, risk_level_blocked, approval_required, rate_limited). Tools that resolve with isError: true are recorded as failures so health metrics reflect reality.

MCP — client & server

@weaveintel/mcp-client@weaveintel/mcp-server@weaveintel/mcp-statsnz

What it does. @weaveintel/cache

Safety, governance & reliability #

This is the layer that makes weaveIntel suitable for production rather than a demo. Most of it works as a default once attached to the runtime — you opt in once, and every agent and tool inherits the behaviour.

@weaveintel/guardrails — input/output & tool-call checks

@weaveintel/guardrails

What it does. A pipeline of checks that run at defined stages of an agent's life. createGuardrailPipeline(guardrails, opts) takes an array of declarative Guardrail rows (each with an id, type, stage and config) and evaluates the applicable ones at each stage. Alongside it the package ships building blocks you can compose into your own guardrails or gates: a DefaultRiskClassifier that scores content risk, confidence and action gates (DefaultConfidenceGate, DefaultActionGate), a CostGuard that stops runs exceeding a budget, and a governance layer for runtime policies. Helpers like hasDeny, hasWarning and getDenyReason let you inspect the GuardrailResult[] that evaluate returns.

How it plugs in. Expose a guardrails slot on the runtime with checkToolCall and checkOutput. The agent loop then consults them automatically — and crucially, fail-closed: if a check throws, the action is denied and audited rather than allowed. This is why putting guardrails on the runtime, not in each agent, matters: you cannot ship an agent that forgets to enforce them.

guardrails.ts
import { createGuardrailPipeline, hasDeny, getDenyReason } from '@weaveintel/guardrails';
import type { Guardrail } from '@weaveintel/core';

// Guardrails are declarative config rows (id, name, type, stage, config).
const guardrails: Guardrail[] = [
  {
    id: 'post-blocklist', name: 'Safety Blocklist',
    type: 'blocklist', stage: 'post-execution', enabled: true, priority: 1,
    config: { words: ['social security number', 'credit card number'], action: 'deny', category: 'safety' },
  },
];
const pipeline = createGuardrailPipeline(guardrails, { shortCircuitOnDeny: true });

// evaluate(input, stage, context?) returns GuardrailResult[]
const results = await pipeline.evaluate(candidateAnswer, 'post-execution', { tenantId: 'acme' });
if (hasDeny(results)) throw new Error(getDenyReason(results));
🛡
Default outcome — fail-closed enforcement

When the runtime carries guardrails, every agent checks tool calls and final output through them. A denial blocks and audits; an erroring check is treated as a denial, never a pass. There are two complementary hooks: the runtime guardrails slot is the uniform, process-wide mechanism, while the per-agent policy.approveToolCall hook lets a single agent apply its own tool-call policy. Both run; either can deny.

Built-in guardrail types & cognitive checks

A Guardrail is a declarative row with a type the evaluator knows how to run. Five types evaluate synchronously today — regex, blocklist, length, schema and custom — and a sixth, model-graded, is reserved for asynchronous model-scored checks. The interesting behaviour lives under custom, whose config.rule selects a named check. Several of these implement cognitive safeguards aimed at reasoning quality rather than content filtering:

custom ruleWhat it checks
input-patternAgreement-pressure ("sycophancy") phrasing in the user prompt — e.g. "just agree", "don't challenge me" — and warns so the model prioritises truth over agreement.
output-patternOverly-validating phrasing in the answer ("you are absolutely right", "100% correct") that may indicate sycophancy.
grounding-overlapLexical overlap between question and answer; warns on weak grounding. Bypassed when the response was backed by tool evidence (then it is grounded by construction).
decision-balanceDevil's-advocate check: decision-style questions ("should I…", "best…", "recommend") should surface counterpoints or trade-offs in the answer.
risk-confidence-gateClassifies the action's risk by verb (read / write / modify / destructive) and applies a risk-scaled confidence threshold via a confidence gate.
aggregate-confidence-gateCombines prior cognitive results into one confidence score, penalised by risk level, and gates on it.

The package also ships DEFAULT_GUARDRAILS — twelve seed rows covering PII redaction, a toxicity filter, two prompt-injection guards, a token budget, a hallucination check, and the six cognitive checks above — as plain typed objects you load into your own store. Cognitive checks emit warn (advisory), not deny: they shape and flag rather than block, and the consuming application decides how to surface or act on a warning.

🧵
How geneWeave uses this

geneWeave consumes these checks from the package rather than re-implementing them. It stores guardrails as DB rows, maps its row types (cognitive_check, content_filter, factuality, budget, redaction) onto the package's custom/rule and built-in types with a small adapter, then runs them through createGuardrailPipeline in both its runtime guardrails slot and its chat pipeline. The sycophancy, grounding and devil's-advocate logic is the package's; geneWeave supplies the configuration (patterns and thresholds as data) and the orchestration that persists evaluations and surfaces warnings.

Current capabilities vs. enterprise hardening

The guardrails available today are fast, deterministic and operator-configurable, which makes them dependable and cheap to run. Their detection is, by design, lexical and rule-based. The following is an honest map of what ships now against what a thorough enterprise deployment would add — useful both for setting expectations and for planning extensions.

What ships today

  • Synchronous regex, blocklist, length and schema guardrails, plus the named custom cognitive rules above.
  • A priority-ordered pipeline that short-circuits on the first deny, with allow / warn / deny decisions and per-check confidence.
  • A runtime guardrails slot consulted fail-closed by the agent loop on every tool call and final output, with denials audited.
  • A built-in verb-based risk classifier and a confidence gate, plus runtime-policy evaluation for cost ceilings, token limits, rate limits and tool restrictions.
  • Twelve-row default seed (PII, toxicity, prompt-injection, token budget, hallucination, six cognitive checks).

@weaveintel/redaction — PII detection & masking

@weaveintel/redaction

What it does. weaveRedactor({ patterns, reversible }) scans text for PII and replaces each match with a token. Built-in pattern types cover email, phone, ssn, credit_card and ipv4; you can add custom regex patterns and an allow-list. With reversible: true it keeps a token→original map so you can restore real values in the model's output after the model has done its work on the masked text. The call is redactor.redact(ctx, text) (context-first) and returns { redacted, detections } where each detection is { type, original, token }. weavePolicyEngine() ships alongside for policy evaluation.

When to use it. Before sending user text to a third-party model when you must not leak PII, and after receiving the response when you want the real values back in the final answer. It pairs naturally with the runtime audit logger's redacting variant so logs never capture raw secrets.

redaction.ts
import { weaveRedactor } from '@weaveintel/redaction';

const redactor = weaveRedactor({
  patterns: [
    { name: 'email',       type: 'builtin', builtinType: 'email' },
    { name: 'phone',       type: 'builtin', builtinType: 'phone' },
    { name: 'credit_card', type: 'builtin', builtinType: 'credit_card' },
  ],
  reversible: true,
});

const { redacted, detections } = await redactor.redact(ctx, 'Email me at a@b.com');
// redacted: "Email me at [EMAIL]"; detections: [{ type:'email', original:'a@b.com', token:'[EMAIL]', start, end }]

@weaveintel/encryption — per-tenant field encryption

@weaveintel/encryption9k+ LOC

What it does. Envelope encryption with a per-tenant key hierarchy. A master key wraps a per-tenant key-encryption key (KEK), which wraps per-tenant data-encryption keys (DEKs), which encrypt individual fields. KMS providers ship for local development plus AWS KMS, Azure Key Vault, GCP KMS and HashiCorp Vault Transit. A weaveTenantKeyManager({ kms, store, audit }) manages the hierarchy; maybeEncryptField / maybeDecryptField apply a field policy; a database proxy encrypts on write and decrypts on read transparently; and you get blind-index search over encrypted columns, key rotation, and hard-shred (crypto-erase by destroying a tenant's keys). Ciphertext uses a self-describing enc:v1:<epoch>:<iv>:<ct> sentinel so the DEK epoch travels with the data and rotation is non-destructive.

When to use it. When you store sensitive customer data and need defence in depth and a credible deletion story. Hard-shred is the key feature for "delete my data" requests: destroy the tenant's keys and the ciphertext is unrecoverable, no row-by-row scrubbing required.

encryption.ts
import { weaveTenantKeyManager, LocalKmsProvider, maybeEncryptField, DEFAULT_FIELD_POLICY } from '@weaveintel/encryption';

const keys = weaveTenantKeyManager({
  kms: new LocalKmsProvider(),    // swap for AwsKmsProvider / AzureKeyVaultProvider / … in prod
  store, audit,
});

const ciphertext = await maybeEncryptField(ctx, keys, DEFAULT_FIELD_POLICY, 'ssn', '123-45-6789');
// → "enc:v1:1:…:…"  (self-describing; decrypts with the matching DEK epoch)

@weaveintel/resilience — rate limits, breakers, retries

@weaveintel/resilience

What it does. The shared reliability pipeline every outbound call flows through. createResilientCallable(fn, options) wraps a function with a token-bucket rate limiter, a circuit breaker, a concurrency limiter and a retry policy, all keyed on an endpoint id so state and signals are shared across every caller of that endpoint. It emits normalized signals on a bus your app can react to, and per-call withOverrides({ rateLimitMode }) lets one call site choose fail-fast while another waits.

When to use it directly. Mostly you don't — providers and the governed tool registry already route through it. Reach for it when you call an external API of your own and want the same protection. The provider defaults are codified once in PROVIDER_RESILIENCE_DEFAULTS so behaviour is consistent across OpenAI, Anthropic and Google; custom providers reuse the constant.

OptionMeaning
endpointThe shared key, e.g. 'openai:chat:gpt-4o' — all callers of this endpoint share breaker/limiter state.
retryRetry policy or options (attempts, base/max delay, jitter).
timeoutMsPer-call timeout enforced by the pipeline.
defaultRateLimitMode'wait' (block until a token frees) or 'fail-fast' (throw immediately).
signalBusBus to emit normalized resilience signals on; defaults to the process-wide bus.
resilience.ts
import { createResilientCallable, PROVIDER_RESILIENCE_DEFAULTS } from '@weaveintel/resilience';

const callApi = createResilientCallable(rawFetchFn, {
  endpoint: 'acme:billing',
  retry: PROVIDER_RESILIENCE_DEFAULTS.retry,     // 2 attempts, 500ms base, 30s cap, jitter
  timeoutMs: 10_000,
  defaultRateLimitMode: 'wait',
});

await callApi(ctx, req);                          // shares breaker state with all 'acme:billing' callers
await callApi.withOverrides({ rateLimitMode: 'fail-fast' })(ctx, req);   // this one won't block

Naming — durability vs reliability

Note the deliberate split from @weaveintel/resilience: resilience is the live per-call guard pipeline (token bucket, breaker, retry), while durability is operational at-least-once machinery that uses resilience internally but addresses a different concern.

When to use it. Around triggers, webhooks, queues and long-running workflows — anywhere a message might be delivered twice or a job might fail and need parking. Idempotency keys keep double-delivery safe; the DLQ keeps failures inspectable rather than lost; health checks feed your orchestrator. geneWeave exposes health snapshots in its admin dashboard built on these. See examples/125-durable-runtime.ts and 126-durable-subsystems.ts.

Observability & evaluation #

You cannot run agents in production you cannot see. This family makes every model call, tool call and agent step visible, attributes cost to it, evaluates output quality, and lets you replay a run after the fact.

@weaveintel/observability — tracing, usage, budgets

@weaveintel/observability

What it does. Provides tracers and trackers that the rest of the framework emits into. weaveConsoleTracer() prints spans for local development; weaveInMemoryTracer() captures them for assertions and inspection. weaveUsageTracker() accumulates token usage; weaveBudgetTracker() raises alerts as spend approaches a cap; weaveTraceGraph() reconstructs the parent/child span tree (with formatTraceGraph to print it); weaveRunTimeline() renders a run as an ordered timeline; and weaveJsonSink() exports spans as JSON for an external system.

How it's wired. Set a default tracer with weaveSetDefaultTracer(...) or attach one to the runtime, and every agent, tool and model call resolves it via weaveResolveTracer(ctx) and emits spans — with token counts, durations and the operation name — without any per-call instrumentation from you. That is the payoff of observability riding the context.

observability.ts
import { weaveSetDefaultTracer, weaveContext } from '@weaveintel/core';
import { weaveInMemoryTracer, weaveTraceGraph, formatTraceGraph } from '@weaveintel/observability';

const tracer = weaveInMemoryTracer();
weaveSetDefaultTracer(tracer);          // now every agent/tool/model call emits spans

const ctx = weaveContext({ userId: 'u_1' });
await agent.run(ctx, { messages: [{ role: 'user', content: '…' }] });

console.log(formatTraceGraph(weaveTraceGraph(tracer.getSpans())));   // parent/child span tree with timings
@weaveintel/live-agents-trace-tools
Trace-inspection tooling specialised for live-agent meshes — visualise heartbeats, ticks, messages and contract evolution over time. See examples/110-live-agents-trace-tools.ts.

@weaveintel/cost-governor — spend control with levers

@weaveintel/cost-governor50 files

What it does. Turns "LLM spend" from a surprise into a managed quantity. A cost ledger (weaveCostLedger, createInMemoryCostLedger, createDurableCostLedger) records every call's cost via computeUsd; weaveCostGovernor and weaveBudgetGate enforce ceilings; and a set of levers reduce cost before you hit them. The levers each address a different waste source:

LeverWhat it does
weaveModelCascadeResolverTry a cheap model first; escalate to a stronger one only when needed.
weavePromptCachingShaperShape requests to hit provider prompt caches (the cache-key hint providers forward).
weaveToolSubsetFilter · weaveIntentRagToolSubsetFilterSend the model only the tools relevant to the turn, cutting prompt size.
weaveToolOutputTruncatorTrim oversized tool outputs before they re-enter the prompt.
weaveHistoryCompactorCompact long conversation history to stay within context and cost budgets.
weaveStaticCostPolicyResolverResolve a cost policy per tenant/task from configuration.

When to use it. As soon as cost is real. The governor gives you a single place to set budgets and a toolbox of techniques to live within them, with a ledger that attributes spend by task so you know where the money goes. geneWeave's cost-by-task dashboard and its sv:cost-gate script are built on this. See examples/103–109.

import { weaveCostGovernor, createInMemoryCostLedger, weaveModelCascadeResolver } from '@weaveintel/cost-governor';

const ledger = createInMemoryCostLedger();
const governor = weaveCostGovernor({ ledger, monthlyUsdCap: 200 });

// Lever: start cheap, escalate only when the task needs it
const resolver = weaveModelCascadeResolver({ tiers: [cheapModel, strongModel] });

@weaveintel/evals — test your prompts & agents

@weaveintel/evals

What it does. A test harness for LLM behaviour. weaveEvalRunner({ model }) runs an eval definition over a set of cases and checks each against assertions. Six assertion types ship: exact, contains, regex, schema (structural), latency_threshold and cost_threshold. The call is runner.run(ctx, definition, cases) — context first.

When to use it. To stop prompt and model changes from silently regressing. Build a corpus of representative cases with expected properties, run it in CI, and gate deploys on the result. Because latency and cost are assertable, an eval can fail not just on a wrong answer but on one that's too slow or too expensive. geneWeave's sv:eval script runs a scientific-validation corpus this way. See examples/09-eval-suite.ts and 30-prompt-eval-optimization.ts.

evals.ts
import { weaveEvalRunner } from '@weaveintel/evals';
import type { EvalDefinition, EvalCase } from '@weaveintel/core';

const runner = weaveEvalRunner({ model });

// Assertions live on the DEFINITION as { name, type, config }.
const def: EvalDefinition = {
  name: 'Capital of France', type: 'model',
  assertions: [
    { name: 'mentions-paris', type: 'contains', config: { substring: 'Paris' } },
    { name: 'cheap-enough',  type: 'cost_threshold', config: { maxUsd: 0.01 } },
  ],
};

// Cases carry an id and an input with messages.
const cases: EvalCase[] = [
  { id: 'fr', input: { messages: [{ role: 'user', content: 'Capital of France?' }] } },
];

const result = await runner.run(ctx, def, cases);   // ctx first
// result → { name, totalCases, passed, failed, avgScore, avgDurationMs, results }
console.log(`${result.passed}/${result.totalCases} passed`, result.results);

@weaveintel/replay — deterministic re-execution

@weaveintel/replay

What it does. Captures a run as a scenario and re-executes it deterministically, so you can debug a production incident, compare two versions on the same input, build golden-case suites, and detect regressions. createReplayEngine(opts?) returns a ReplayEngine whose replay(ctx, runLog) re-runs a recorded RunLog and returns a ReplayResult; the package also exports scenarios, golden cases, comparison runs and regression detection (the engine, scenario, golden, comparison and regression modules).

replay.ts
import { createReplayEngine } from '@weaveintel/replay';

const engine = createReplayEngine();
const result = await engine.replay(ctx, recordedRunLog);   // re-executes a captured RunLog deterministically
console.log(result);   // → ReplayResult: what matched, what diverged

When to use it. When "it behaved differently in production" needs a reproducible answer. Record the run, replay it locally against a code change, and compare — or freeze a known-good run as a golden case and fail CI if behaviour drifts. Live-agent runs have their own replay path (replayLiveAgentsRun) since their state evolves over many ticks. See examples/62 and 78-kaggle-replay-roundtrip.ts.

Platform & domain #

The remaining packages are the platform substrate — durable storage, event triggers, skills, prompt management, UI streaming, collaboration — plus a few domain-specific packs that show how vertical capabilities are built on the core.

@weaveintel/persistence — one adapter, many databases

@weaveintel/persistence

What it does. A single persistence adapter abstraction (createPersistenceAdapter) with concrete backends for in-memory, SQLite, Postgres, Redis, MongoDB, CosmosDB and cloud NoSQL. Durable subsystems across the framework (cost ledgers, idempotency stores, trigger stores, OAuth state, audit logs, budget enforcers) write through this so persistence is configured once and reused.

When to use it. When you move from in-memory prototypes to a real database. Pick the backend that matches your stack; the durable variants of other packages accept an adapter, so switching from SQLite locally to Postgres in production is a configuration change, not a rewrite. See examples/119-sqlite-e2e.ts.

@weaveintel/triggers — event-driven activation

@weaveintel/triggers

What it does. Turns external events into framework actions. A trigger dispatcher (createTriggerDispatcher) routes manual, cron, webhook, queue and change events to bindings that start a workflow, run an agent, or call an outbound webhook. Trigger definitions live in a store with backends for SQLite, Postgres, Redis, MongoDB and DynamoDB, and a durable rate limiter guards against floods.

When to use it. Whenever something other than a user request should wake your system — a nightly cron job, an inbound webhook from a SaaS, a row change in a database, a message on a queue. Triggers are the bridge between the outside world and agents/workflows, and they're how live-agent meshes receive external events. See examples/99-db-driven-triggers.ts and 124-ambient-agent.ts.

Skills & capability packs

@weaveintel/skills@weaveintel/capability-packs

What skills do. A skill bundles a set of tools, a tool-policy key and activation logic into a named capability an agent can switch on for a turn. createSkillRegistry holds them, activateSkills turns on the relevant ones for a context, collectSkillTools gathers their tools, and createSkillTelemetry records which skills fired. Activating a skill can also swap the agent's tool policy, so "research mode" and "billing mode" carry different permissions.

What capability packs do. Where a skill is in-process, a capability pack is operator-shippable: a JSON manifest of database rows — a workflow plus its handler kinds, tools, triggers and prompts — that you export from one weaveIntel instance and install into another. The package is schema-agnostic; an app declares its row kinds through a PackInstallAdapter. This is how a vertical capability moves between deployments without code.

skills.ts
import { createSkillRegistry, activateSkills, collectSkillTools } from '@weaveintel/skills';

const skills = createSkillRegistry();
skills.register({ key: 'billing', tools: billingTools, toolPolicyKey: 'billing-policy' });

const active = await activateSkills(ctx, skills, { intent: 'refund' });
const tools  = collectSkillTools(active);   // hand to the agent for this turn

@weaveintel/prompts — versioned, governed prompts

@weaveintel/prompts

What it does. Treats prompts as managed assets rather than string literals. It supports many prompt kinds (template, few-shot, structured, chain, router, judge, optimizer), versioning with draft/active status, typed variables, reusable framework sections and fragments, output contracts, model-compatibility metadata, A/B experiments and variants, and a resolver that selects the right version at runtime. createTemplate / createSafeTemplate build templates; records can be hydrated from the database with createPromptDefinitionFromRecord.

When to use it. @weaveintel/ui-primitives

When to use it. When your front end is more than a text box — when an agent should return a table the user can sort, a chart, a form to fill, a citation to click, or an approve/deny card. Security & defaults — what you get for free #

Throughout this guide, "default outcome" callouts noted behaviour you get without writing it. Here they are in one place — the safety posture a weaveIntel application has by construction once you attach a runtime and use the governed forms. None of this is bolted on; it lives in the loop, the registry and the context.

Safe egress

All outbound HTTP flows through a hardened, SSRF-checked client that blocks cloud-metadata and RFC-1918 targets, re-validates every redirect hop, and caps time and size.

Secret hygiene

Credentials resolve through runtime.secrets, never process.env in leaf code — so vaults and per-tenant overrides plug in without touching call sites.

Six-stage tool governance

The governed registry checks enabled → circuit → risk level → approval → rate limit before executing, then audits the outcome. Every call, every time.

Fail-closed guardrails

Agents consult runtime guardrails on each tool call and final output. A denial — or an erroring check — blocks and audits. You can't ship an agent that skips them.

Audit everywhere

Agent runs, tool invocations, denials, and live-agent actions emit weaveAudit records automatically, with a redacting logger option so logs never carry raw secrets.

Per-tenant encryption

Envelope encryption with isolated keys per tenant and crypto-erase deletion — enabled by a single master key, transparent to app code via the DB proxy.

Deny-by-default identity

An identity has no access until a rule grants it; delegation is scoped; ACL-aware retrieval keeps users to what they may see.

Budgets & cost levers

Per-execution and per-tenant budgets are enforceable, with a ledger attributing spend and levers (cascade, caching, tool-subset, compaction) to live within them.

Resilience defaults

Provider and tool calls share rate-limit + circuit-breaker + retry state per endpoint, with consistent PROVIDER_RESILIENCE_DEFAULTS and normalized signals.

Replayable runs

Traces, workflow checkpoints and live-agent run logs let you replay a run deterministically to debug, compare versions, or guard against regressions.

Humans hold the keys

In a live-agent mesh, agents cannot self-promote, self-grant capabilities, or bind external accounts — enforced as typed errors, not convention.

Sandboxed execution

Model-generated code runs under resource limits and a network/filesystem policy, optionally in an OS-level container.

The through-line

Every one of these works because cross-cutting concerns ride the ExecutionContext and its WeaveRuntime rather than living in your call sites. Attach the runtime once, use the governed registry and the live/agent loops, and the framework enforces the posture uniformly. Your code stays about the product.

End-to-end: a woven mesh #

This is the composite the framework is built for — a long-running live agent that uses governed tools, retrieval, guardrails and observability together, persisted in a state store and driven by a heartbeat. It shows how the layers you've read about snap together. The shape follows the runnable examples/comprehensive-live-agents-workflow.ts and the equity-analyst mesh in examples/120-equity-analyst-mesh.ts; the snippet below is an original, accurate composition of the verified APIs.

woven-mesh.ts
import {
  weaveContext, weaveRuntime, weaveToolRegistry, weaveTool,
  envSecretResolver, weaveInMemoryPersistence,
} from '@weaveintel/core';
import { weaveInMemoryTracer } from '@weaveintel/observability';
import { createPolicyEnforcedRegistry, InMemoryToolPolicyResolver, DEFAULT_TOOL_POLICY, noopAuditEmitter } from '@weaveintel/tools';
import { weaveRetriever, weaveEmbeddingPipeline } from '@weaveintel/retrieval';
import { weaveFakeEmbedding, weaveFakeVectorStore } from '@weaveintel/testing';
import {
  weaveLiveAgent, weaveLiveAgentPolicy,
  createActionExecutor, createHeartbeat, weaveInMemoryStateStore,
} from '@weaveintel/live-agents';
import { weaveOpenAIModel } from '@weaveintel/provider-openai';

async function main() {
  // ── 1. Runtime carries the cross-cutting defaults: tracer, secrets,
  //      persistence and fail-closed guardrails. Everything below inherits them.
  // The guardrails slot is implemented directly — the agent loop calls these.
  const runtime = weaveRuntime({
    tracer: weaveInMemoryTracer(),
    secrets: envSecretResolver(),
    persistence: weaveInMemoryPersistence(),
    guardrails: {
      async checkToolCall(ctx, schema, args) {
        return schema.riskLevel === 'destructive'
          ? { allow: false, reason: 'destructive tools disabled' }
          : { allow: true };
      },
      async checkOutput(ctx, text) {
        return text.includes('API_KEY')
          ? { allow: false, reason: 'leaked secret' }
          : { allow: true };
      },
    },
  });

  // ── 2. Knowledge: ingest docs, expose retrieval as a governed tool.
  const embed = weaveFakeEmbedding(), vstore = weaveFakeVectorStore();
  const ctx0 = weaveContext({ tenantId: 'acme', runtime });
  const ingest = weaveEmbeddingPipeline({ embeddingModel: embed, vectorStore: vstore });
  await ingest.ingestDocument(ctx0, { id: 'kb-1', content: 'Refunds are processed within 5 business days.' });
  const retriever = weaveRetriever({ embeddingModel: embed, vectorStore: vstore, topK: 3 });

  const rawTools = weaveToolRegistry();
  rawTools.register(weaveTool({
    name: 'search_kb', description: 'Search the knowledge base',
    parameters: { type: 'object', properties: { q: { type: 'string' } }, required: ['q'] },
    riskLevel: 'read-only',
    execute: async ({ q }, ctx) => {
      const { chunks } = await retriever.retrieve(ctx, { query: q });   // retrieve() → { chunks, query }
      return chunks.map(c => c.content).join('\n');
    },
  }));

  // ── 3. Governance: wrap the registry so every call is gated + audited.
  const tools = createPolicyEnforcedRegistry(rawTools, {
    resolver: new InMemoryToolPolicyResolver(new Map([
      ['search_kb', { ...DEFAULT_TOOL_POLICY, rateLimitPerMinute: 30 }],
    ])),
    auditEmitter: noopAuditEmitter,        // a constant, not a call
    resolutionContext: { agentPersona: 'support' },
  });

  // ── 4. The live agent — same option names as weaveAgent, plus a policy bundle.
  const { handler } = weaveLiveAgent({
    name: 'support-coordinator',
    role: 'support',
    model: weaveOpenAIModel('gpt-4o-mini', { apiKey: process.env.OPENAI_API_KEY }),
    tools,
    systemPrompt: 'You answer support questions using the knowledge base. Cite what you find.',
    maxSteps: 20,
    policy: weaveLiveAgentPolicy({ auditEmitter: { async emit(ev) { console.log(`[audit] ${ev.toolName} → ${ev.outcome}`); } } }),
  });

  // ── 5. Persist a mesh/agent/contract + an inbound question, then tick.
  const store = weaveInMemoryStateStore();
  // (saveMesh / saveAgent / saveContract / saveMessage / saveHeartbeatTick as in the live-agents section)

  const heartbeat = createHeartbeat({
    stateStore: store, workerId: 'w1', concurrency: 1,
    attentionPolicy: { key: 'support', async decide() { return { type: 'ProcessMessage', messageId: 'm1' }; } },
    actionExecutor: createActionExecutor({ handlers: { 'agentic.react': handler } }),
    now: () => new Date().toISOString(),
  });

  // One wake of the mesh: the agent retrieves (governed + audited),
  // the guardrails check the tool call and the answer, the tracer records spans.
  const { processed } = await heartbeat.tick(weaveContext({ tenantId: 'acme', runtime }));
  console.log(`processed ${processed} tick(s)`);
}

main().catch(console.error);

Read it as a stack: the runtime (1) supplies tracing, secrets and fail-closed guardrails to everything derived from its context; retrieval (2) becomes a tool; the governed registry (3) gates and audits that tool; weaveLiveAgent (4) runs the ReAct loop per tick with a policy bundle; and the heartbeat (5) wakes the agent on schedule against a persisted mesh. Swap the in-memory store for Postgres and the fake embedding for a real one and the same code is production-shaped. For the full multi-mesh version — cross-mesh bridges, account binding, MCP delivery and compression — run examples/comprehensive-live-agents-workflow.ts.

If this guide is the map, geneWeave is the territory — every layer you've read about is wired together here, in code you can run and read.

How to run it. From a fresh clone, one script takes you to a running app at http://localhost:3500:

bash
# installs deps, builds packages, writes .env, seeds the DB, and starts the server
./scripts/start-geneweave.sh

# useful flags
./scripts/start-geneweave.sh --no-start   # install + build + seed only
./scripts/start-geneweave.sh --prod       # run via deploy/server.ts
./scripts/start-geneweave.sh --rebuild    # force a clean rebuild

The script is idempotent and safe to re-run. The apps/geneweave/src tree reads like an index of this guide. A few highlights:
