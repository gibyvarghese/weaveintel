08 · Plain language

Name things by what they do

Every label a person reads — nav items, field names, buttons — is written in plain words, even in the admin Builder. System internals (the API name, the data type, the storage key) never surface as UI copy. Admins are people too; they shouldn't need the schema to read the screen.

SYSTEM INTERNAL — don't show WHAT THE USER SEES
Prompt Fragment → Building block
key → Shortcut
Variables (JSON) → Fill-in values
enabled: true → Active · On
Orchestration → Workflows
Routing → Request handling
Governance → Rules & limits
webhook config → Connected apps
Do
  • Label fields by their purpose: "What it's for," "Fill-in values."
  • Buttons name the result: "Save," then a "Saved" toast.
  • Keep the technical token only where it's genuinely needed — as a quiet mono help line, never the label.
Don't
  • Expose data types or storage keys as UI labels.
  • Use insider nouns like "fragment," "orchestration," "A2A protocol."
  • Show raw values (a bare 1, an em-dash) where a word would read clearer.
